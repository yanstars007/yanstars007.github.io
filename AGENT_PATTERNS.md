# 🤖 通用智能体 Agent 范式总结

本文档总结了基于通用智能体 (Universal Agent) 的开发思路与设计模式，特别是结合 OpenClaw / Hermes 等系统的实践。

## 1. 范式转移：从“写代码”到“教做事”

### 📉 传统模式 (Hard-coded / 机械臂)
- **实现方式**：LangChain / Dify 等编排工具，画流程图。
- **逻辑**：硬编码的 `if-else` (如果 A 发生 -> 调工具 B -> 执行动作 C)。
- **痛点**：
  - 极度死板，遇到未知路径直接崩溃。
  - 维护成本高，业务逻辑一变就得改代码、重新部署。

### 📈 智能体模式 (Context-Driven / 高材生)
- **实现方式**：通用大脑 (LLM) + 知识文档 (Markdown)。
- **逻辑**：
  - **SOUL.md (宪法)**：定义性格、红线、价值观。
  - **SKILLS.md (技能书)**：定义专业能力（如“怎么部署 Docker"），按需加载。
  - **MEMORY.md (经验值)**：记忆上下文和历史反馈。
- **优势**：
  - **自然语言编程 (Declarative)**：用文档定义边界，用大脑处理过程。
  - **极速组合**：加载不同技能即可变身不同专家 (SRE/客服/程序员)。
  - **高容错率**：具备推理能力，能处理流程之外的意外情况。

---

## 2. 核心架构：自治进化闭环 (Autonomous Evolution Loop)

这是一个实现从报警到自我提升的 **数字运维专家 (Digital SRE)** 标准工作流。

### 🔄 闭环流程
```text
[报警源] (Webhook) 
   ↓
1. 接收与研判 (Ingest & Triage)
   查规则 (Rules) + 查记忆 (Memory) → 区分“误报”与“新问题”
   ↓
2. 执行 (Execute)
   ├─ [已知/误报] → 自动处理 (忽略/记录/修复)，不打扰人类。
   └─ [新问题]    → 升级人工 (Escalate)，请求决策。
   ↓
3. 反馈与学习 (Feedback & Learn)
   人类反馈 (对/错/忽略) → Agent 捕获意图。
   ↓
4. 自我进化 (Evolve)
   Agent **自行修改规则文件** (patch monitoring-rules.md)。
   下次同样情况直接自动处理。
```

### 💡 关键点：声明式规则 (Declarative Rules)
规则不再是代码里的逻辑，而是 Agent 维护的 Markdown 文档。
Agent 通过读写这个文档来实现“经验积累”。随着时间推移，规则库越来越厚，人工干预越来越少。

---

## 3. 实施步骤 (Action Items)

1. **定义身份**：编写 `AGENTS.md`，确立红线和原则。
2. **配置技能**：编写 `SKILLS.md`，定义具体操作 (如 Webhook 接收、终端命令)。
3. **初始化规则库**：创建 `monitoring-rules.md`，填入初始已知规则。
4. **打通 Webhook**：让外部系统能通过 HTTP POST 触发 Agent。
5. **闭环测试**：发送测试报警 -> 确认人工反馈 -> 验证规则自动更新。
